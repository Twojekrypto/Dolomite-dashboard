# Lessons Learned — Dolomite Dashboard

## Editing & Version Control
- **Always verify edits applied**: After a cancelled or interrupted edit, re-check the file — the edit may have been reverted.
- **Check for duplicate markup**: When modifying HTML in a large file, old markup can persist alongside new markup if edits don't target the exact right lines.
- **Unbalanced Flex Containers**: Be extremely careful when adding blocks to existing `display:flex` layout toolbars. A single stray opening/closing `</div>` tag will detach siblings from the flex flow, destroying alignment ("rozjechany interfejs"). Always cross-check the target edit chunk perfectly matches tag depth.
- **ZAWSZE podawaj komendy push**: Agent nie może `git push` (macOS Keychain blokuje). Po KAŻDEJ zmianie podaj userowi gotową komendę do wklejenia w terminal, np.: `git -C ~/Desktop/Draft/Dolomite\ website add -A && git commit -m "msg" && git push origin main && git push origin main:master`

## Deployment & Verification
- **GH Pages cache**: Always add a unique timestamp/version parameter to URLs when verifying deployed changes (e.g., `?v=abc123&t=<timestamp>`).
- **Wait for GH Actions**: GitHub Pages deployment can take 30-60 seconds. Don't assume changes are live immediately after push.

## CSS & Layout
- **`text-align: center` on `<td>`**: Affects ALL child content including flex containers and breakdown text. When centering some columns, target them individually (`:nth-child(2)`, `:nth-child(3)`, etc.) rather than using ranges like `:nth-child(n+2)`.
- **Column alignment independence**: Supply APR column with breakdowns should stay `text-align: left` while other data columns can be centered.
- **Tooltip UX standard**: Informational hover text should use the dashboard glass tooltip pattern from oDOLO Claimer Breakdown (`#float-tip` / matching `#unified-tooltip` styling), not native browser `title` bubbles or old inline `.tooltip-bubble` visuals. In Earn, every explanation (`Using Replay`, `BORROW`, `VERIFIED`, APR breakdowns, exact values, reward pills, and address hovers) must route through body-level `data-tip` / `addr-tooltip-wrap` and render with the Earn `#unified-tooltip.earn-tooltip` skin so tables do not clip tooltips and the UX stays consistent.
- **Earn tooltip cursor contract**: Earn explanations can show the unified tooltip, but they must not force `help`, text/I-beam, or any other cursor shape just because a hover explanation exists. Use `cursor: default` for non-interactive tooltip triggers (`data-tip`, verification chips, APR lines, detail rowline values), keep `pointer` only for real clickable controls, and do not add tooltip copy to copy/explorer icons in expanded Earn detail rows.
- **Earn compact USD subvalues should stay quiet**: Do not add hover tooltips to the small `≈ $...` subvalue under `Current Balance` in Supply Assets. It is supporting display text, not an explanation surface; exact value hovers add noise.
- **Earn Past & Routed header is a full-row toggle**: `Supply Assets` and `Borrow Positions` keep the explicit collapse button, but `Past & Routed Assets` intentionally behaves like a full-width accordion header. Keep the chevron visual consistent, avoid nested button double-toggles, and browser-verify header click plus Enter/Space keyboard toggles after changing this section.
- **Earn verification label hierarchy**: Main Earn rows should show one user-facing verification status (`Verified`, `Pending`, `Fallback`, etc.). Do not show `Replay Verified` beside `Verified`; replay is the verification method and belongs in expanded details as `Verification method: History replay`.
- **New UX search bars require quick clear**: Every gold/new-design search bar for addresses, labels, tokens, or tx hashes must include a small visible `X` clear control when it has input. The clear control should reset the value, dispatch the normal `input` event so filters rerender, and keep focus in the search field. Do not spend time retrofitting the old design unless explicitly requested.
- **Tables must not resize on search/filter**: In the new UX dashboard, visible table shells must keep stable width and height when filters, searches, or periods reduce result counts. Use fixed table layout/colgroups and fill short pages with inert spacer rows (or equivalent fixed body height), then verify with browser `getBoundingClientRect()` before shipping.
- **Assets table viewport**: Dolomite Assets should show a stable 10-row table viewport, then scroll inside the table body/wrap. Do not let the table shell grow to the full asset count or shrink after filters.
- **Assets detail rows follow oDOLO detail design**: Expanded Dolomite Assets rows should use the `Top oDOLO Exercisers` detail pattern: gold-accented parent row, gradient detail-row background, inner bordered panel with left gold rail, eyebrow/title header, and framed content area.
- **All Chains filter icon contract**: The `All Chains` dropdown option must mirror the trigger button's full globe icon (circle, horizontal line, and curved vertical meridian). The TVL Token Composition filter is the reference for the select-all row styling and icon treatment; reuse that pattern in Assets.

## Data Display
- **Non-oDOLO rewards bug**: Some tokens (e.g., USD1) have rewards from external protocols (WLFI) that aren't oDOLO. These must be captured as yield sources and controlled by the Yield toggle, not ignored.
- **Stablecoin formatting**: Stablecoins should show fewer decimals (1) since their values are close to $1. Volatile tokens need more precision (6 decimals).
- **Earn routed collateral is not withdrawn**: In Earn, assets parked inside a borrow subaccount as collateral must not be labeled `WITHDRAWN`. Use a position-state label such as `In Borrow Route`, styled differently from data-quality chips like `Verified`; use `Exited Position` only for assets that are no longer active supply/collateral/borrow-route positions.
- **Open borrow collateral yield is not normal closed yield**: If Earn replay shows yield on collateral still routed inside an open borrow position (for example WETH marked `In Borrow Route`), show it as open/borrow-route collateral yield and keep it separated from ordinary `Total Yield Earned`. It can become normal realized/closed-route yield only after the route is closed or the collateral leaves the borrow route.
- **Borrow-position collateral yield must use replay ledger**: In Borrow Positions details, never calculate routed collateral yield from `par * supplyIndex - par`. That is market-index math and can overstate a specific user's open route. Use the per-account replay `liveYield` / open-collateral yield so Borrow Positions and Past & Routed show the same wallet-level basis.

## Browser Subagent Tips
- **Give explicit wait times**: Subagents need clear wait durations for data-loading pages. "Wait 8 seconds" works better than "wait for data".
- **Specific click targets**: "Click the ASSETS tab" can fail if the subagent clicks the wrong element. Include position hints or element descriptions.
- **Layout Alignment Checks**: After making layout/flexbox changes to the UI, always spawn a browser subagent instructed to navigate to the specific component and visually verify alignment (left vs right, vertical centering) vs the design requirement *before* committing. Do not blindly assume your HTML injected correctly.

## GitHub Pages Deployment
- **Active production repo is `Twojekrypto/Dolomite-dashboard`**: Treat `vedolo-dashboard` as legacy/retiring. In this local checkout, `dolomite-dashboard` is the remote for production pushes; do not deploy new dashboard changes to old `origin` unless explicitly requested.
- **Always push production to `master`**: The Dolomite Dashboard GitHub Pages site serves from `master`. When pushing from this checkout, use the `dolomite-dashboard` remote, e.g. `git push dolomite-dashboard master`.
- **macOS Keychain blocks background `git push`**: When deploying from a headless AI agent background process, macOS security (`securityd`) blocks the Keychain prompt and causes a hanging password loop. Never run `git push` via `run_command` if it triggers a credential prompt; always instruct the user to run it explicitly in their physical Terminal.

## Table Column Restructuring
- **Check CSS `nth-child` rules**: When removing/adding table columns, CSS `nth-child` selectors silently override inline styles. Always audit ALL `nth-child` references after changing column count.
- **`data-col` on th**: If a th contains interactive elements (search input, dropdowns), remove `data-col` to prevent sort triggers from accidental clicks.
- **Sort indicators only on the active column**: Do not show inactive `⇅` arrows in table headers. For Dolomite tables, sortable-but-inactive columns should look like normal headers; only the current sorted column shows `▲` or `▼`. If using a `.sort-arrow` span, leave it empty for inactive columns instead of writing a placeholder glyph. For tightly aligned numeric detail tables, do not render an empty sort span at all; an invisible span can still reserve width and make headers look left-shifted relative to right-aligned values.
- **Non-sortable action columns need the full table contract**: When moving an action like oDOLO `Details` into its own column, add a non-sortable `<th>` without `data-sort`, update detail-row `colspan`, spacer/placeholder rows, and narrow header listeners to `th[data-sort]`. Verify in the browser that action buttons moved out of the wallet cell and still expand the detail row.
- **Earn expandable rows should advertise Details consistently**: Supply Assets, Borrow Positions, and Past & Routed Assets should all use a separate non-sortable `Details` column with the same `Details/Hide` pill pattern as the Assets page. When adding it, update colgroups, detail-row colspans, spacer rows, mobile/card CSS, and every `nth-child` alignment rule together.
- **Expanded detail tables need explicit column contracts**: For nested/detail tables like oDOLO `Wallet exercise history`, use `table-layout:fixed` plus a `<colgroup>` with widths matching the main table's numeric/action columns. Leave only the text/date column flexible. Without this, the browser distributes extra width across every column and numeric data drifts away from the right-side table rhythm.
- **Verify detail table alignment at real desktop width**: After changing nested table columns, measure header and first-row `getBoundingClientRect()` values in a served page. Compare numeric/action columns against the related main table, and also check mobile because media rules often change nested tables to block/card layouts.
- **veDOLO holder locks should be user-facing "Positions", not "NFT"**: The backend may expose `nft_count` / `token_ids`, but the UI should label these as `Positions`, `Position Count`, and `Position ID`. Reserve NFT terminology only for technical/debug contexts so users understand these are veDOLO lock positions, not collectible NFTs.
- **Keep holder collapsed rows clean**: In veDOLO holder tables, collapsed wallet rows should show only `Wallet`, shortened address, copy, DeBank, aggregate DOLO/vote/position count, and a `Details` action. Do not inline long position/token ID lists next to the wallet label; move position IDs into the expanded detail row.
- **Position detail rows should copy the oDOLO Details pattern**: Use a separate non-sortable `Details` column, row-click toggle, open-row highlight, `Hide/Details` pill with chevron, and a nested table styled like `Top oDOLO Exercisers`. Include explicit `colgroup`, correct `colspan`, sortable detail headers with active-only arrows, and prevent copy/DeBank clicks from toggling the row.
- **Do not confuse detail-panel accents with row hover**: The steady gold stripe on an expanded veDOLO/oDOLO detail panel communicates that the parent row is open and should remain visible. Row-hover strips inside the nested table are separate per-row feedback and should be fixed with selector scope, not by removing the panel stripe.
- **Nested table row hover must use direct-child selectors**: Global table hover strips should be written like `.tbl > tbody > tr:hover > td:first-child::before`, not `.tbl tbody tr:hover td:first-child::before`. Broad descendant selectors bleed into expanded detail tables and can make several nested position rows look hovered when only one row is under the cursor.
- **Expanded position table column order should follow oDOLO**: In veDOLO position breakdowns, keep the first real data column as the flexible date/text column (`Unlock Date`), then put right-aligned numeric metrics (`DOLO`, `Vote Weight`, `Remaining`), and keep compact technical identifiers (`Position ID`) at the far right. This matches the `Top oDOLO Exercisers` detail rhythm and avoids leading with a technical chip before the user-facing timing/value context.
- **veDOLO Flow should stay paired on desktop**: Keep `Recent DOLO Locks` and `Recent veDOLO Unlocks` as two side-by-side panels on wide desktop for comparison. Use explicit colgroups, compact active-only sort arrows, `Wallet / Date` as the flexible column, right-aligned DOLO/Lock metrics, and a small icon-only `Tx` column at the end. On narrower screens the panels can stack, but the column contract should remain readable with internal horizontal scroll if needed. Keep source labels inline beside the wallet: `Direct`, `via oDOLO`, and compact `Protocol` for protocol-routed fallback locks, with full meaning in the tooltip/title if needed.
- **Flow source chips need custom table-safe tooltips**: For compact labels such as `Direct`, `via oDOLO`, and `Protocol`, prefer a fixed-position JS tooltip over native `title` or in-row CSS bubbles. Tables and flow panels can clip hover content, and the label semantics matter enough to show the explanation in the dashboard's own UX.
- **Interactive charts should use the local chart contract**: Bars, donut slices, and similar chart marks should be real focusable controls with hover/focus preview and click-to-pin active state. Put the selected metric in a compact DOM focus panel, keep hover tooltips minimal and positioned near/above the mark so they do not cover the data being inspected, and verify computed sizes/cursors/overflow in the browser QA report.
- **Compact KPI charts use the same interaction contract**: veDOLO mini charts such as `Withdrawal Behavior` should not be passive inline bars. Render each metric as a focusable button row with hover/focus preview, click-to-pin state, active-row dimming, and a compact focus panel. Bar fills must be block-level elements (`display:block`) so widths/heights render reliably.

## CI/CD & Data Pipelines
- **Always cache API receipts**: Scripts that fetch on-chain tx receipts grow linearly slower over time. Always use a cache file + GH Actions `actions/cache@v4` for incremental syncing.
- **git pull --rebase -X theirs**: For automated JSON data commits, use `-X theirs` to auto-resolve merge conflicts (our freshly generated data is always newer/correct).
- **Timeout headroom**: Set `timeout-minutes` to at least 2× expected runtime to account for RPC slowness and data growth.
- **⚠️ GitHub Actions 15-minute timeout destroys cache**: If a data sync job (like `update-odolo-flows.yml`) relies on `actions/cache@v4` to save state at the end, a harsh timeout (e.g. `timeout-minutes: 15`) will CAUSE the cache to NEVER SAVE if the initial full-sync takes longer than 15m! The job enters a death-loop: runs 15m → cancels → drops state → next run starts from scratch. Always set a very generous `timeout-minutes` (e.g., `120`) for pipelines doing bulk historical block fetching via RPC, so they can finish the full sync once and save their structural cache.
- **Long canonical-history workflows should checkpoint, not fail**: Baseline refreshes for Berachain, Mantle, and selected Arbitrum hot wallets can exceed one polling window. The workflow should save runtime cache and mark the refresh incomplete without running ledger/audit/commit steps; the next scheduled run continues from cache instead of showing a red failure and restarting work.
- **Botanix EARN onboarding**: Botanix EARN starts at block `690000`, margin `0x003Ca23Fd5F0ca87D01F6eC6CD14A8AE60c2b97D`, and currently has five markets (`WETH`, `pBTC`, `USDC.e`, `stBTC`, `oUSDT`). Prefer `https://rpc.botanixlabs.com` for full `eth_getLogs` ranges; `https://rpc.ankr.com/botanix_mainnet` is a useful fallback but can reject large ranges, and `https://rpc.botanixlabs.dev` has timed out during local audits.

## UI Overflow & Dropdowns
- **`overflow:hidden` clips dropdowns**: Parent containers with `overflow:hidden` will clip absolutely-positioned child elements (like dropdown menus). Use `overflow:visible` instead when the container has interactive dropdowns.
- **⚠️ `.earn-hero-outer` MUST be `overflow:visible`**: This element contains the chain selector dropdown. Using `overflow:hidden` (for gradient border glow containment) will clip the dropdown menu. Instead, use `clip-path:inset(-2px round 20px)` on the `::before` pseudo-element to contain the glow effect without affecting child overflow. Also needs `z-index:200` so dropdown appears above sibling elements below.
- **CSS `:empty` to hide unused containers**: Instead of `display:none` + JS toggle, use `:empty` pseudo-class to auto-hide containers that haven't been populated yet. Avoids timing issues with JS initialization.

## ⚠️ MANDATORY: JavaScript Verification of CSS/UI Changes
- **ALWAYS verify CSS changes via browser JS console** before considering any UI fix done. Use `getComputedStyle(el).paddingLeft`, `el.getBoundingClientRect()` etc. to confirm the actual computed values match your intent.
- **Never trust inline style values alone** — CSS rules with `!important` or higher specificity can silently override inline styles. Always run `getComputedStyle()` to see what the browser actually renders.
- **Check for `!important` overrides first**: Before changing any CSS padding/margin/position, search for `!important` rules that might target the same element (e.g. `thead th:first-child` overriding a specific th). Root cause analysis saves hours of trial-and-error.
- **Measure alignment with `getBoundingClientRect()`**: When aligning elements, use JS to get exact `.left` positions of both elements rather than guessing pixel values.
- **After removing HTML elements, search for ALL JS references**: Removing an HTML element (e.g. `#m-exit-updated`) without fixing the JS that does `getElementById('m-exit-updated').textContent = ...` causes a null reference crash that can silently break entire sections downstream (e.g. the Recent Early Exits table).
- **⚠️ grep ALL IDs before removing elements**: When converting a dropdown to pills (e.g. `risk-dropdown` → inline `risk-pills`), grep for ALL references across click handlers / toggle / close logic. Missing even 1 stale `getElementById('risk-dropdown')` will break OTHER dropdown handlers that try to close it (because null.classList throws).

## Dolomite E-Mode & Risk Overrides
- **Never assume global liquidation ratio applies to all accounts**: Dolomite uses E-Mode (Automatic Efficiency Mode) for correlated asset pairs (e.g., stablecoin↔stablecoin). E-Mode uses a lower liq ratio (111.11% vs 115%) and zeroes margin premiums. This must be queried via RPC `getAccountRiskOverride()` on the `defaultAccountRiskOverrideSetter` contract.
- **Subgraph doesn't expose per-account risk overrides**: The `MarginAccount` entity has no `marginRatioOverride` field. E-Mode overrides live on the smart contract level only.
- **Use web3.py with proper ABI, not raw eth_call**: For struct parameters like `Account.Info`, use `w3.eth.contract(abi=...).functions.X().call()` — manual ABI encoding of struct tuples is error-prone.
- **⚠️ E-Mode RPC: use `user.id`, NOT `effectiveUser.id`**: `getAccountRiskOverride((owner, number))` requires the actual on-chain owner (vault/proxy = `user.id`), not the wallet behind it (`effectiveUser.id`). For vault accounts like iBGT isolation mode, `effectiveUser` returns marginRatioOverride=0 while `user` correctly returns 0.333. Bug pattern: all vault/proxy accounts in e-mode will have incorrect (inflated-risk) health factors if `effectiveUser` is used.
- **⚠️ NEVER use bare `except: pass` on RPC calls**: Silent exception swallowing on rate-limited RPCs caused 170+ Ethereum E-Mode positions to be missed (2/347 → 173/347). Multicall3 with 200 calls → 413 "Payload Too Large" → individual fallback → 429 rate limit → ALL silently failed. Fix: smaller batch sizes (50), retry with exponential backoff (1-3s), inter-batch delays, and proper error counting.

## Data Pipeline Resilience
- **Silent chunk failures on RPC → empty chain data**: `eth_getLogs` scanning 15M+ blocks on Berachain with rate-limited RPCs causes chunks to silently fail, producing `total_transfers=0`. Always count failed chunks and add defensive fallback: if fresh scan returns 0 but cache has data, use cached data.
- **Default period must show data**: If a table uses a configurable period (7d/30d/ALL), default to a period that shows existing data. oDOLO Exercises had latest data from 2025-08-07 but defaulted to 7 days → permanently empty. Default to ALL.
- **⚠️ Smart contracts can have MULTIPLE method IDs for the same action**: The oDOLO Vester contract has TWO exercise methods (`0xa88f8139` original, `0xf3621c90` newer). The pipeline only tracked the first, missing 51% of recent exercises. Always check for method ID variants when a data pipeline suddenly shows stale data.
- **⚠️ Cross-chain bridge transfers inflate flow data**: When the same address bridges tokens between chains (ETH↔Bera), per-chain flow calculations count it as outflow on source and inflow on destination. Must neutralize same-address opposing flows across chains — otherwise holders appear as massive sellers. Use `neutralize_cross_chain_flows()` pattern: detect opposing flows on multiple chains, cancel the min overlap.
- **⚠️ Bridge mints/burns invisible to flow calc**: `calculate_flows()` skips transfers from/to 0x0 (mints/burns). Bridges use burn→0x0 on source and mint←0x0 on destination, making one side invisible to neutralization. Fix: `calculate_bridge_flows()` extracts mint/burn amounts separately, augments raw_flows before neutralization runs, then delta is applied back to original flows. This way bridge patterns become visible without polluting overall flow semantics.
- **⚠️ Fallback data hides real sell-offs**: When balance RPC returns 0, a holders fallback was overriding with stale cached data. This made addresses that sold ALL tokens appear to still hold them. Fix: only apply fallback for addresses where RPC actually FAILED (network error), not for legitimate zero balances.
- **⚠️ ERC20 `balanceOf` batch results must be ABI-valid**: `balanceOf(address)` should return a 32-byte ABI-encoded uint256 (`0x` + 64 hex chars). Short values like `0x0`, missing batch items, or per-item RPC errors are failed calls, not trustworthy zeros. Retry those addresses individually and only fall back to transfer-ledger reconstruction when the balance call truly fails.
- **⚠️ Leveraged borrow positions inflate yield**: The Earn tab yield formula `yield = currentWei − netFlow` breaks for leveraged accounts. All-account netflow includes swap/borrow/transfer deltas from sub-accounts (margin accounts), producing garbage yield (e.g., $4.6M for a $12K supply). Root cause: Balance = account 0 only (RPC), but Netflow = ALL accounts (subgraph `effectiveUser`). Internal transfers between the same user's accounts are invisible in the filtered transfer queries. **FIX**: Query `marginAccount { accountNumber }` from the subgraph for deposits, withdrawals, and ALL transfers (including internal same-user, fetched separately without `toEffectiveUser_not` filter). Build a separate `acct0Netflows` object tracking only flows affecting account 0. For borrow markets, use `acct0Netflows` instead of `netflows`. Priority: non-borrow → all-account netflow > snapshot > dash. Borrow → snapshot > acct0 netflow > dash.
- **⚠️ Column visibility ≠ calculation method**: The `hasAnyYield` check (which controls yield column visibility via `no-yield` CSS class) must NOT filter by borrow market status. The borrow market filter only determines which yield *calculation method* to use per-row (all-account netflow vs acct0 netflow vs snapshot), not whether yield data *exists*.
- **⚠️ Dolomite multi-account architecture**: Each address can have multiple "accounts" (account numbers) within DolomiteMargin. The RPC `getAccountBalances(addr, accountNum)` only returns balances for ONE specific account number. Borrow positions live on different account numbers (large numbers like `32240150...`). The subgraph `deposits` entity has `marginAccount { accountNumber }` — use this to distinguish account-0 deposits from margin-account deposits. Internal transfers (same-user, between accounts) don't appear in the regular transfer queries (filtered by `fromEffectiveUser != toEffectiveUser`) — fetch them separately with `fromEffectiveUser` only, then check `fromMarginAccount.accountNumber` / `toMarginAccount.accountNumber`.
- **⚠️ NEVER clamp yield to 0**: Supply yield CAN be genuinely negative if liquidation losses or unfavorable swap deltas on account 0 exceed accrued interest. Clamping to `0n` hides real losses from the user, which is dishonest. Instead, display negative yield in red with a `-` sign prefix. The `earn_formatAmount` function already handles negative BigInt values correctly (line 21177: `isNeg = value < 0n`). When using `yieldSign` prefix, always use the three-way ternary: `totalYield > 0n ? '+' : totalYield < 0n ? '-' : ''`, and pass `abs(yield)` to formatAmount to avoid double minus.
- **⚠️ NEVER use parseFloat for wei arithmetic**: Subgraph `amountDeltaWei` returns human-readable decimals (e.g. `"3452.742715"`). Use `decStrToBigInt(str, decimals)` to parse to BigInt directly. `parseFloat` loses precision beyond 15-16 digits (IEEE 754), leading to drift in accumulated netflow for whale positions with many transactions. Example: 50 tx × 10K tokens → ~992 wei error. For 10K+ tx → errors could reach $0.01+. The `addFlow`/`addAcct0Flow` functions now use native BigInt arithmetic with zero precision loss.
- **⚠️ Track ALL balance-changing events**: Dolomite has 7 event types that change balances: deposits, withdrawals, transfers, trades, liquidations, **vaporizations**, and interest accrual (implicit in supply index). Missing ANY type causes incorrect netflow → incorrect yield. Vaporizations are debt absorption events: `solidEffectiveUser` gains `heldToken`, `vaporEffectiveUser` loses collateral. Query both `solidEffectiveUser` and `vaporEffectiveUser` with `marginAccount.accountNumber` for acct0 filtering.
- **⚠️ History heuristic: avoid aggressive yield caps**: The guard `if (totalBorrowed > 0n && yield > totalDeposited / 10n) → zero out` incorrectly hides legitimate multi-year yields (which can exceed 10% of principal). Only keep the `totalWithdrawn > totalDeposited` guard (detects borrow-funded withdrawals where netflow formula is unreliable).

## External Rewards (oDOLO + MERKL WLFI)
- **MERKL v3/rewards API**: Use `https://api.merkl.xyz/v3/rewards?chainIds={id}&user={addr}` to get exact WLFI earned. Filter `campaignData` by reason keys starting with `Dolomite_`. The reason key format `Dolomite_{tokenAddr}` allows per-token attribution.
- **oDOLO metadata API**: `https://api.dolomite.io/liquidity-mining/odolo/metadata` returns per-token weekly allocation (`allChainWeights`). Calculate per-position oDOLO as `(userSupply / totalSupply) × weeklyAlloc × weeksActive`.
- **⚠️ WLFI blacklist**: Some wallets (e.g., `0x5be9...`) are blacklisted in MERKL campaigns. They appear in the `params.blacklist` array of campaign data and receive 0 WLFI despite supplying the eligible token.
- **⚠️ Per-position attribution**: MERKL rewards are wallet-wide, NOT per-position. Use `perToken` matching (reason key → collateral token address) to correctly attribute rewards to specific borrow positions. This prevents double-counting when a wallet has multiple positions.
- **⚠️ Summary card pattern**: Dynamic summary stats should only appear when data exists. Use `if (data > threshold) return;` early exit + `getElementById()` dedup check to prevent duplicate cards on re-render.

## Grid Layout & Inline Style Overrides
- **Inline `style.gridTemplateColumns` ALWAYS wins over CSS classes**: When JS sets `el.style.gridTemplateColumns = 'repeat(5,1fr)'`, it overrides any CSS class like `.cols-6 { repeat(3,1fr) }`. Solution: never use inline grid overrides — manage grid layout ONLY through CSS classes toggled via JS.
- **Symmetric card layouts**: For 6 cards use `repeat(3,1fr)` → 3+3 rows. For 7 use `repeat(4,1fr)` → 4+3. Centralize layout management in a single function (`earn_updateSummaryCardLayout`) that reads child count and applies the right class.

## Fixed-Position Dropdowns in Overflow Containers
- **`position:absolute` + `overflow:auto` = CLIPPED dropdowns**: When a parent container has `overflow:auto` (e.g., `.table-container`), any `position:absolute` child extending beyond the container bounds will be clipped. This is especially visible when the table has few rows (e.g., after filtering).
- **Solution: `position:fixed` + JS positioning**: Switch dropdowns from `position:absolute` to `position:fixed` and calculate `top`/`left` from `getBoundingClientRect()` of the trigger button. Use a helper like `positionPopoverFixed(triggerEl, popoverEl)`.
- **Clamp to viewport**: Always clamp fixed-position dropdowns to viewport bounds: `if (left + width > window.innerWidth - 8) left = window.innerWidth - width - 8`.

## Portfolio Value Calculation
- **Include borrow equity in Portfolio Value**: The EARN tab previously only summed supply-only assets. For users with borrow positions (margin trading), the true portfolio value = supply assets + (collateral − debt). Update the Portfolio Value card in `earn_updateSummaryDebt()` when lending positions arrive.

## Earn Tab — Hide Dust Toggle
- **⚠️ Hardcoded filters ignore toggle state**: The `earn_renderResults` function had a hardcoded `a.usdValue >= 0.01` filter that always removed dust from the supply table, regardless of the `earn_showDust` toggle. Pattern: always grep for ALL hardcoded thresholds when a filter toggle "doesn't work" — the toggle logic may be fine, but the render path may bypass it.
- **⚠️ Dust threshold = $1, not $0.01**: User expects positions below $1 to be considered dust, not $0.01. The threshold must be consistent across: main supply/borrow filter, history render, withdrawn assets, inline withdrawn section, and dust count calculation.
- **⚠️ Borrow-only addresses show "No deposits"**: When `balances.length === 0` (no supply on account 0), the code early-returned before the subgraph borrow data could be injected (borrow fetch runs in `earn_fetchNetflow()` which was started AFTER the balance check). Fix: when no supply balances, call `earn_fetchNetflow()` first to check for borrows, then proceed with borrow-only render path if found.
- **⚠️ Borrow-funded withdrawals inflate Withdrawn Assets yield**: The `yield = currentWei - netFlow` formula produces false multi-million dollar "yields" when an address borrows tokens (e.g., 3M USD1) against collateral and then withdraws them. Since `currentWei = 0` (withdrawn) and `netFlow = deposits - withdrawals` is very negative (withdrawals >> deposits because excess was funded by borrowing), the formula computes a huge positive yield. Fix: if `totalWithdrawn > totalDeposited`, the excess was borrow-funded → zero out yield. Also clamp when `totalBorrowed > 0` and yield exceeds 10% of deposits. Note: `earn_borrowMarkets` only tracks OPEN borrows (status=OPEN), so closed/repaid borrows aren't detected — must use the withdrawal-vs-deposit heuristic instead.

## Earn Audit Workflow
- **Always preserve wallet identity on timeout rows**: The live audit runner must include `address` even for `eval_timeout`, otherwise timeout retries cannot target the real remaining cohort and the summary overstates unresolved work.
- **Use `full pass -> timeout retry -> tiny final retry`**: For large EARN cohorts, the best quality/speed tradeoff is one full pass for the whole unresolved input, then a focused retry only for timeout categories, and finally a tiny last retry for the few timeouts that survive the first retry.
- **Do not count `missing_position` as active failure**: A wallet that no longer has the market live should be separated from active verification stats. Final `X/X verified` must use `active = cohort - missing_position`, not total cohort rows.
- **Measure merged final state by latest result per wallet**: When combining a full pass with retries, compute the final report by overwriting older rows with the newest row for the same wallet. Raw row counts from intermediate files can be misleading.
- **Timeout retries can convert the majority of unresolved cases**: For hard Arbitrum EARN cohorts, many apparent failures are just slow replay cases. A focused timeout retry can turn most of them into `replay_verified` without changing verification logic.
- **Fast future audits should split easy and hard work**: The next iteration of `audit_earn_asset.py` should support a built-in `fast first pass + focused retries` mode so new asset audits finish sooner without relaxing correctness standards.
- **Normalize audit categories from raw row state, not stale saved labels**: Hidden collateral rows with `replay-ledger/public-* + pre_snapshot_carry` or tiny verified drift can be false negatives if the summary trusts the original stored `category`. Recompute the effective category from `positionKind`, resolved source/status, and verification drift when summarizing results so older batches and new assets both benefit from classification fixes.
- **Separate blocking mismatches from dust/informational tail**: In EARN asset audits, low-value hidden-collateral dust and tiny snapshot remnants should not be mixed with real blockers. Summaries should report both raw unresolved counts and a stricter `blockingRealNonVerified` number so follow-up work targets true data issues first.
- **Treat borrow-only residue as non-active for supply-yield audits**: If a market ends up as sub-dollar supply plus real borrow exposure, that is not an active supply-yield failure. Count it as non-active in audit summaries instead of inflating the unresolved active-wallet cohort.
- **Never overwrite replay indexes with partial null refreshes**: Live replay refreshes and subgraph supplementation can return only one side of `supplyIndex/borrowIndex`. Merge index maps field-by-field and preserve existing non-null values; otherwise a later partial refresh can silently erase borrow verification and create false `snapshot_only` fallbacks on mixed supply/borrow markets.
- **Hidden-collateral-only markets must still enter total-yield history scope**: If `earn_fetchTotalYield` only receives visible active markets, collateral parked inside borrow accounts can miss both snapshot history and replay-backed yield, creating fake `hidden_collateral_other` blocker rows. Always union `earn_collateralPositionData` into the needed market set for history loading and resolved-yield refreshes.
- **Use merged/forensic audit modes instead of hand-stitching retries**: `audit_earn_asset.py merge-live`, `extract-live`, and `forensic-live` should be the default post-processing steps after a full pass. They keep the latest row per wallet, build retry cohorts without losing metadata, and turn the remaining tail into a concrete blocker report instead of ad-hoc JSON analysis.
- **Split pure hidden-collateral gaps from mixed hidden+visible overlap**: If a row looks like `hidden_collateral_other` but the verification map still shows positive `actualSupplyWei`, that is a different blocker class from a pure hidden-collateral gap. The audit should flag it separately (for example `mixed_hidden_visible_overlap`) so future assets immediately show whether the bug is hidden-subaccount coverage or a mixed visible+collateral replay reconciliation issue.
- **Never use `value || ''` when auditing market id `0`**: In the live audit runner, `String(pos.marketId || '')` turns `marketId = 0` into an empty string and silently hides the real visible position for assets like Arbitrum `WETH`. Use a nullish check (`value === undefined || value === null ? '' : String(value)`) so market `0` rows are captured correctly.
- **Separate hidden collateral coverage gaps from hidden collateral math gaps**: If a `hidden_collateral_other` row has positive actual collateral, no visible position, no borrow position, and no replay-backed total-yield data at all, that is a coverage problem, not a generic collateral mismatch. Flag it explicitly (for example `hidden_collateral_coverage_gap`) so future asset audits show that the blocker is missing replay coverage rather than bad yield arithmetic.
- **Split mixed visible+collateral+borrow mismatches out of generic `has_data_other`**: When a row has visible supply, collateral, and borrow positions at the same time, plus replay verification data on all three sides, a remaining mismatch is a mixed-position reconciliation bug, not an unknown bucket. Classify it explicitly (for example `mixed_position_overlap_mismatch`) so future audits can tell mixed subaccount reconciliation issues apart from ordinary netflow fallback drift.
- **Count audit blockers by severity, not just by pattern name**: Once audits learn new explicit blocker classes, a low-severity coverage gap should not be counted the same way as a medium/high active mismatch. Final blocker counts and extracted retry cohorts should treat `low/info` tails as informational even if they have a new pattern name, and reserve `blocking` for the medium/high rows that really need forensic follow-up.
- **Never use `value || ''` for live `marketId` handling either**: The same falsy-coercion bug that broke the audit runner can also corrupt the actual EARN runtime. `String(pos.marketId || '')` silently drops Arbitrum `WETH` (`marketId = 0`) from current-position reconciliation, active-market warmup, and history loading. Always normalize market ids with a nullish check (`value === undefined || value === null ? '' : String(value)`).
- **Preserve borrow-account overrides through replay refreshes and merges**: After current balances or snapshot supplementation identify which subaccounts have open borrows, every later replay re-summarization must carry that same borrow-account set forward. If refresh/supplement/merge paths recompute replay summaries without the override, mixed wallets with visible supply plus collateral+borrow subaccounts can fall back to `netflow-fallback / mismatch` even though the replay is otherwise correct.
- **Feed explicit borrow-account routing into both replay and current-balance bucketing**: Hidden collateral should not be classified only by the late heuristic "positive balance on an account that happens to have borrow right now." Carry forward explicit borrow-account numbers from replay/subgraph `borrowPositions`, annotate current subgraph snapshots with `borrow / hidden-collateral / visible-supply`, and bucket current balances from that routing so hidden collateral stays attached to the right subaccount even when live RPC coverage is partial.
- **Trust replay-ledger verification before generic `has_data_other` fallback in the auditor**: Some live rows can still have `verificationData.status = unverified` because `snapshotIncomplete` stayed set, even though the resolved runtime result is already `replay-ledger + verified` and the remaining drift is literally `1 wei`. Audit normalization should classify those rows as verified from resolved replay state before falling through to generic `has_data_other`, otherwise assets like `WEETH` show false blocker tails despite trusted final yield.
- **Trust exact hidden-collateral matches when local interest-ledger data is already verified**: Some GMX-style rows can stay in `hidden_collateral_other` even though `calc.hasData = true`, `calc.verificationStatus = verified`, `calc.trustedForTotal = true`, and all `supply/collateral/borrow` diffs are exactly zero. That is an audit false negative caused by missing propagated resolved metadata, not a real coverage gap. Normalize those rows to verified so future GMX asset audits do not overstate blocker tails.
- **Treat `interest-ledger-live-balance-adjusted` as the same trusted replay family as other verified interest-ledger methods**: Once replay is verified and only the final live wei drift is being applied from current balances, downstream resolved metadata, history summaries, and audit views must propagate that path exactly like `interest-ledger` and `interest-ledger-override`. Leaving it out of trusted-method sets creates fake unverified tails even though the replay result is already canonical.
- **Treat GMX/V2 `dGM` cohorts as live-first, not netflow-first**: Static audits for markets like `WBTC-USDC-GMX-V2`, `WETH-USDC-GMX-V2`, and `gmETH` can show `100% snapshot-only / no_netflow`, but live replay/public verification still resolves most wallets. For these markets, the real tail tends to collapse into low-severity hidden-collateral coverage gaps after one timeout retry, so future audits should go straight from static unresolved extraction to live pass + timeout retry instead of over-interpreting the static `no_netflow` result.
- **Keep canonical `VERIFIED` narrower than helpful evidence**: Historical snapshot ledgers, public-netflow reconciliation, pre-snapshot carry, and live-balance drift adjustments are all valuable evidence, but they are not the same thing as exact replay verification. Preserve them as `historical`, `inferred`, `coverage_incomplete`, or `fallback` paths and reserve canonical `VERIFIED` for exact replay/current reconciliation with full coverage.
- **Move operational knobs to config, not strict logic**: URLs, worker counts, polling intervals, timeout windows, and checkpoint cadence belong in `config/earn_live_defaults.json`; strict classification rules and blocker patterns do not.
- **Treat audit classification as unit-test territory**: `normalize_live_row_category`, `parse_live_row_pattern`, `summarize_live_results`, and config loading should always have fast unit coverage, plus a generated `LIVE_AUDIT_JS` syntax check in the local/CI check runner before merge.
- **Live EARN audits must target the routed Earn page**: After the dashboard route migration, `/index.html` loads the DOLO shell and does not expose `switchView`/`earn-address`, so live audits return a false `switchView unavailable` / `unknown` cohort. Default audit URLs must point at `/earn/?cb=...`, and a quick smoke run should confirm categories are real (`replay_verified`, `hidden_collateral_verified`, etc.) before interpreting a full asset report.
- **Incremental canonical-history plans must self-heal stale selected wallets**: A selected hot wallet can be below the global manifest baseline if it was added after the baseline moved. Do not leave it as `staleTrackedAddresses` with zero tasks; send stale selected histories through the full backfill path to the target block, then future delta applies can keep them fresh.

Standard operacyjny to `full pass -> timeout retry -> merge -> forensic -> combined report -> true-tail diff explain`, żeby kolejny ruch naprawczy wynikał z gotowego raportu, a nie ręcznego sklejania JSON-ów.
**Reguła na przyszłość:** Sam lock plikowy nie wystarcza przy live auditach. Przed startem kolejnego chained marketu sprawdź też, czy nie ma już aktywnego `audit_earn_asset.py live` uruchomionego poza runnerem, bo dwa równoległe batch-e na tym samym Chrome/localhost łatwo mieszają środowisko i dają mylące wyniki.
**Reguła na przyszłość:** Exact zero-diff z `fresh` canonical history, które kończy jako `fallback` albo `inferred`, nie powinno lądować w worku `has_data_other`. To nie jest strict verified, ale też nie jest prawdziwy blocker matematyki. Oznaczaj takie przypadki osobnym, niskoseverity patternem diagnostycznym, żeby true tail nie był sztucznie napompowany.
**Reguła na przyszłość:** Nie wyłączaj verified-ledger fetch w live audycie, jeśli chcesz mierzyć realne zachowanie runtime. Cache można wyczyścić, ale sam fetch powinien zostać aktywny, inaczej audit sztucznie zaniży historyczny baseline i wyprodukuje fałszywe `pending`/coverage gapy dla hidden collateral.
**Reguła na przyszłość:** `strictStatus = inferred` z `rawStatus = pre_snapshot_carry` i `canonicalHistoryCoverageStatus = fresh` nadal nie jest strict `verified`, ale jest wystarczająco dobrym historycznym baseline'em, żeby zasilić runtime total-yield. Odrzucanie takich wpisów zmusza lookup do gorszych fallbacków albo pustych wyników, mimo że kanoniczny ledger już ma lepszą wiedzę.
**Reguła na przyszłość:** Timeout retry nie musi używać tych samych ustawień co full pass. Dla retry ogona warto osobno tunować liczbę workerów, bo mniejszy cohort często czyści się szybciej przy trochę większej równoległości bez mieszania całego chained runu.
**Reguła na przyszłość:** Świeżość canonical history w runtime powinna być liczona per-chain, a nie stałą liczbą bloków dla wszystkich sieci. Globalne `64` bloki oznacza różne opóźnienia na Ethereum, Berachain i Arbitrum; dla dashboardu lepszy jest jawny budżet czasu (np. ok. 3h przeliczony na bloki dla danej sieci), żeby normalny cron-lag nie dawał fałszywego `pending verify`.
**Reguła na przyszłość:** Dla ciężkich sieci nie uruchamiaj scheduled canonical history na całym znanym uniwersum walletów, jeśli celem jest UX lookupów. Używaj bounded hot-wallet selection z priorytetowymi pinami dla adresów zgłoszonych przez użytkownika; pełny backfill zostaw jako osobny, świadomy etap operacyjny.
**Reguła na przyszłość:** Pierwszy baseline canonical history musi mieć cache runtime w GitHub Actions: event shardy, `.progress`, `.repair-plans`, `.incremental-plans` i katalog historii chaina. Bez tego timeout albo przerwanie joba kasuje godziny skanu i kolejny run zaczyna od zera.
**Reguła na przyszłość:** Workflow canonical history dla chaina bez skonfigurowanego `start_block` albo bez znanych EARN adresów powinien kończyć się kontrolowanym skipem, nie failure. Dopiero po pojawieniu się danych w snapshots/netflow oraz poprawnym `start_block` można wymagać strict coverage.
**Reguła na przyszłość:** W workflowach, które commitują dane na `master`, pojedyncze `git pull` przed `git push` nie wystarcza przy wielu równoległych cronach. Remote może przesunąć się między pullem a pushem; używaj pętli `pull --rebase -X theirs && push` z kilkoma próbami, a dopiero po udanym pushu zdejmuj stash z cache/progress files.
**Reguła na przyszłość:** Materializacja z one-pass event shardów musi brać końcowy target z manifestu (`globalToBlock`/`scanRanges`), nie z najwyższego fizycznego shard file. Sparse chain może nie mieć zdarzeń w końcowych chunkach, a wtedy max nazwa pliku zaniża `lastScannedBlock` mimo pełnego pokrycia skanu.
**Reguła na przyszłość:** Repair taski z pre-slicowanym `--address-file` nie mogą dostać dodatkowo `--start-index/--end-index`. Plik adresów jest już shardem, więc drugie cięcie robi puste workery m2+ i fałszywie zatrzymuje naprawę coverage.
**Reguła na przyszłość:** Status materializacji/repair musi być target-aware. `completed` z poprzedniego niższego `targetBlock` nie może spełniać nowszego planu, bo runner pominie potrzebny repair i UI dalej będzie widział stale canonical history.
**Reguła na przyszłość:** Repair progress key musi identyfikować nie tylko target block i numer workera, ale też konkretny repair cohort/plan. Dwa plany dla tego samego targetu z inną liczbą adresów nie mogą mieć tych samych `repair-tBLOCK-mXofY`, bo status z poprzedniego planu fałszywie zamknie nowszy repair i zostawi część walletów stale.
**Reguła na przyszłość:** Incremental canonical-history plan nie powinien blokować świeżego cyklu przez orphan history files, których nie ma już w aktualnym snapshot/netflow universe chaina. Raportuj je jako `orphanedHistoryCount`, ale nie wrzucaj do `staleTrackedAddressCount`, `newAddressBackfill` ani tracked delta apply bez jawnego selection file.
**Reguła na przyszłość:** Delta apply musi tolerować historię świeższą niż `deltaFromBlock - 1`, ale jeszcze starszą niż `targetBlock`. Jeśli delta scan pokrywa cały zakres od baseline, taki wallet można bezpiecznie dostemplować do targetu albo dopisać tylko eventy po jego własnym `lastScannedBlock`; traktowanie go jako stale zatrzymuje coverage po częściowym/naprawczym przebiegu.

### Earn final reveal guard
**Wzorzec błędu:** Finalny render Earn może rzucić wyjątek po pierwszym częściowym renderze, np. przez helper formatowania istniejący w innej zakładce (`fmtUsd`), ale nie w scope Earn. Jeśli exception wydarzy się przed `earn_finishLookupLoadingUi`, użytkownik widzi wieczną kartę ładowania mimo że dane są już policzone.
**Reguła na przyszłość:** Finalny etap lookupu Earn musi mieć `try/finally`, które zawsze zdejmuje `loading-gate` dla aktualnego `runId`. Po zmianach w karcie podsumowania trzeba testować adres z borrow route i verified yield w przeglądarce oraz sprawdzić konsolę na `ReferenceError`/`TypeError`, nie tylko `node --check`.
**Reguła na przyszłość:** Jeśli ten sam market jest jednocześnie widoczną supply pozycją i collateralem w otwartej borrow route, `interestMeta.earnYield` zawiera także live `openCollateralYield`. `Total Yield Earned` musi odejmować ten open-collateral yield i pokazywać go osobno jako `Borrow Route Yield`; do totalu wraca dopiero po zamknięciu trasy albo wyjściu collateral.

### Earn verified ledger cache
**Wzorzec błędu:** Jeśli `data/earn-verified-ledger` jest ignorowany i workflow buduje tylko `--existing-addresses`, publiczny cache pozostaje pusty. Runtime wtedy wraca do pobierania dziesiątek dużych snapshotów EARN, co na Ethereum potrafi zamienić lookup w minutowe ładowanie.
**Reguła na przyszłość:** Dla chainów, które mają przyspieszać UX, workflow musi robić pierwszy seed verified ledger i commitować wybrany katalog przez `git add -f`. Małe chainy mogą używać `build_earn_verified_ledger.py --all-addresses`, ale ciężkie chainy typu Arbitrum/Berachain/Mantle powinny używać bounded hot-wallet selection przez `--address-file`, żeby nie publikować dziesiątek tysięcy plików. W lookupu Earn finalny ekran nie powinien czekać na snapshot fallback; używaj ledger/replay jako fast-path, a snapshoty traktuj jako opcjonalne tło diagnostyczne.

### Arbitrum Earn full live audit
**Wniosek z audytu:** Pełny chained live rerun dla Arbitrum powinien być oceniany po finalnym combined report, nie po surowych kategoriach z pełnego passu. Rerun z 2026-05-02 domknął 71/71 marketów z `0` failed, `0` final blockers i `0` final informational rows; surowe pozostałości typu `eval_timeout` mogą nadal istnieć w agregatach, ale jeśli nie wchodzą do finalnego true tail, nie są blockerem UX/danych.
**Reguła na przyszłość:** Po zakończeniu full-chainowego audytu zapisz liczby końcowe (`completedMarketCount`, `failedMarketCount`, final blockers, final informational, timeout markets) i osobno planuj tylko prawdziwy retry tail. Dla tego przebiegu 5 WETH `eval_timeout` domknęło się po targeted slow retry z dłuższym budżetem (`maxWaitMs=240000`, `lateReplayGraceMs=120000`), a finalny status całego Arbitrum ma `0` timeout markets, `0` blockerów i `0` informational rows. Nie podbijaj timeoutów globalnie dla full passu; dodawaj osobny preset/override dla małych trudnych cohortów.

### Multi-chain Earn live audits
**Wzorzec błędu:** Live-audyt może wyglądać jak masowe `missing_position`, jeśli helper przeglądarkowy przełącza widok na zły chain. To szczególnie zdradliwe, bo wynik ma `ok: true`, ale `canonicalHistory.coverageStatus = missing` mimo istniejących lokalnych plików historii dla audytowanego chaina.
**Reguła na przyszłość:** Każdy live-audyt musi wybierać chain z input payload (`input.chain`), nie wartość hardcoded. Po przerwaniu live passu nie wolno uznać istniejącego pliku wynikowego za kompletny bez sprawdzenia `completed >= inputCount`, bo snapshoty częściowe są zapisywane inkrementalnie i służą tylko do resume.

### Earn live audit verdicts
**Wniosek z audytu Berachain:** Surowe kategorie z live passa (`eval_timeout`, `missing_position`, `no_data`, `timeout_other`) są diagnostyką przebiegu, a nie same w sobie blockerem danych. Pełny Berachain rerun z 2026-05-03 domknął 41/41 marketów z `0` failed, `0` final blockers i `0` final informational rows, mimo że agregaty nadal zawierały mały raw diagnostic tail po etapach live/retry.
**Reguła na przyszłość:** Raporty full-chain live muszą pokazywać osobno `auditVerdict`/`finalTailTotals` i `rawDiagnosticCounts`. Decyzję o poprawności danych podejmuj po finalnym combined report (`finalBlockingTailCount`, `finalInformationalTailCount`, failed markets), a raw diagnostic counts traktuj jako informację do tuningu timeoutów albo obserwacji edge-case'ów.
**Reguła na przyszłość:** Jeśli live row ma `lastReplayError` typu `Subgraph replay timed out`, klasyfikuj go jako timeout, nie `no_data`, żeby trafił do retry. Jeśli `positionKind = missing`, ale widoczny wiersz marketu ma `VERIFIED`, ufaj finalnemu wierszowi UI i nie zostawiaj fałszywego `missing_position` w raw diagnostics.
**Reguła na przyszłość:** Finalny `informationalRows` tail w chained live audit nie powinien od razu zostawiać całego chaina na `WARN`. Jeśli nie ma blockerów, runner powinien automatycznie wyciągnąć informacyjny tail i odpalić mały `targeted-slow-retry`; Mantle WMNT pokazał, że chwilowe `Failed to fetch` dla dust/pre-snapshot-carry potrafi przejść na `verified_nonstandard` po wolniejszym retry bez luzowania reguł audytu.
**Reguła na przyszłość:** Launch JSON z GitHub Actions musi mieć identyfikator aktualnej sesji/runu i tylko PID-y z tej samej sesji mogą być uznawane za żywe albo terminowane. Cache z poprzedniego runnera może zawierać PID, który na nowej maszynie należy już do innego procesu; bez `runnerSessionId` cleanup może błędnie zabić obcy proces i wywalić workflow.
**Reguła na przyszłość:** Ścieżka `allow-checkpoint-incomplete` musi być testowana jako normalny stan operacyjny, nie tylko przez `py_compile`. Długie canonical-history workflowy mają po timeoutach zapisać checkpoint i zakończyć się zielono, więc błąd cleanupu typu `NameError` nie może zmieniać kontrolowanego checkpointu w czerwony run.
